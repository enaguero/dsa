# Notes ‚Äî Long-Form Study Material

This directory holds the deep, prose-style references for the repository. Where [`problems/`](../problems/) is "do the exercise," `notes/` is "understand the theory." Everything here is plain Markdown that GitHub renders directly and that Pandoc compiles to PDF via the eisvogel template (see [`docs/BUILDING.md`](../docs/BUILDING.md)).

The top-level [project README](../readme.md) covers both tracks at a glance.

## Using `ram_model.md`

The document is dense by design but has explicit entry points for different audiences. The document's own ¬ß1.1 ("How to use this document") maps these in detail; here is the short version.

### Pick a reading path

- **First-time reader / coming from problem solving.** Start with **¬ß1.2 "A 5-minute quickstart"** ‚Äî Python `two-sum` example, the three patterns that cover most code you'll write, a "will-it-fit?" table. Then read **¬ß6 ‚Üí ¬ß7 ‚Üí ¬ß11 ‚Üí ¬ß14.9 ‚Üí ¬ß27** in that order; that gives you the working vocabulary in about an hour.
- **Want the theory.** Read Parts I and II in order. Roughly one undergraduate algorithms course's worth of background, compressed.
- **Analyzing your own code.** Skip to Part III. The recipe is in **¬ß13** (worked example first, then the procedure); loop patterns in **¬ß14**; recurrence techniques in **¬ß¬ß16‚Äì18**; amortized, space, randomized in **¬ß¬ß19‚Äì21**.
- **Looking up a specific algorithm.** Part IV is the catalog: ¬ß23 search, ¬ß24 sort, ¬ß25 graph, ¬ß26 DP, ¬ß27 recurrence cheat sheet.
- **Beyond the basic model.** Part V (¬ß¬ß28‚Äì33): memory hierarchy, external memory, cache-oblivious, parallel models, lower bounds, online/competitive analysis.

### Practice as you go

Each Part ends with a short **"Test your intuition"** sidebar (3‚Äì4 quick conceptual checks) whose answers are in **¬ßA**. When you want longer, working-out problems, **¬ßB** holds **60 worked exercises** tagged `[easy] / [med] / [hard]`, distributed across the five Parts:

| Annex section | Topic | Count |
|---|---|---|
| ¬ßB.1 | RAM model (Part I) | 6 |
| ¬ßB.2 | Asymptotic analysis (Part II) | 12 |
| ¬ßB.3 | Mechanics ‚Äî loops, recurrences, amortized, space, randomized (Part III) | 22 |
| ¬ßB.4 | Catalog ‚Äî sort/search/graph/DP (Part IV) | 12 |
| ¬ßB.5 | Beyond RAM ‚Äî external memory, parallel, lower bounds, competitive (Part V) | 8 |

Solutions are in **¬ßC**, structured so the solution to `B.k.j` is at `C.k.j`. Always attempt the exercise first; consult the solution to verify.

## Building the PDFs

Quick reference (full prerequisites are in [`docs/BUILDING.md`](../docs/BUILDING.md)):

```bash
# Build ram_model.pdf (the default SRC) ‚Äî full ~108-page reference.
make pdf

# Build ram_model_essentials.pdf ‚Äî ~28-page subset for first-time readers.
make essentials

# Build any other note.
make pdf SRC=cheatsheet

# Build every note in notes/*.md.
make all

# Build the Beamer slide decks (notes/slides/ram_model_part*.md).
make slides

# Live-rebuild on save (needs fswatch: `brew install fswatch`).
make watch
```

Output PDFs land in `build/`. The `build/` directory is gitignored ‚Äî only the markdown sources are committed.

### Two-track build: full reference vs. essentials

`ram_model.md` is a single source that compiles into two PDFs:

- **`build/ram_model.pdf`** ‚Äî the full ~108-page reference. Pandoc reads the markdown as-is.
- **`build/ram_model_essentials.pdf`** ‚Äî a ~28-page subset with the deep material stripped out: quickstart, asymptotic notation, growth hierarchy, loop patterns, recursion-tree method, Master Theorem core, hidden-costs cheat sheet, recurrence catalog, and the glossary. Just enough to read code, predict complexity, and pass a junior interview.

The mechanism is a Pandoc Lua filter (`templates/strip-advanced.lua`) that drops every fenced div tagged `::: {.advanced}`. To exclude a section from the essentials build, wrap it in markdown:

```markdown
::: {.advanced}

## 9. Properties and Theorems

The detailed manipulation rules go here‚Ä¶

:::
```

Default is "include in both builds" ‚Äî you only need to tag the deep material. Cross-references from kept sections into advanced ones (e.g. "see ¬ß17.5") will dangle in the essentials build by design ‚Äî they become pointers to the full reference.

## How figures work

The figures embedded in `ram_model.md` are PNG files committed under `notes/figures/`, produced from matplotlib scripts in `scripts/figures/`. Committing the PNGs lets GitHub render the markdown without anyone needing a Python environment; rebuilding is only required when a script changes.

```bash
# Regenerate every notes/figures/*.png from scripts/figures/fig_*.py.
# Needs `uv` (brew install uv) ‚Äî matplotlib is pulled in transiently.
make figures
```

The build pipeline uses Pandoc's `--resource-path=.:notes` flag (see `Makefile`) so that image references in the markdown can be written relative to the markdown file itself (`figures/foo.png`) and resolve correctly both on GitHub and during PDF compilation.

## Adding a new figure

1. Write `scripts/figures/fig_<your_slug>.py`. Import the shared palette and helpers from `scripts/figures/_style.py` so the figure matches the document's typography and colors. Save with `save(fig, "your-slug")` ‚Äî the helper writes `notes/figures/your-slug.png` at print-quality DPI.
2. `make figures` to regenerate (this runs `scripts/figures/generate_all.py`, which picks up every `fig_*.py` in the directory).
3. Embed the figure in the markdown with a descriptive caption:
   ```markdown
   ![Caption that names what the figure shows and what to notice.](figures/your-slug.png)
   ```
4. Commit both the script and the PNG.

A few constraints to be aware of when writing figure scripts:

- Matplotlib's mathtext does **not** speak `\le` / `\ge`; use `\leq` / `\geq` instead.
- `set_aspect("equal")` on a wide table-style layout will silently break it; use the default aspect.
- Keep total physical figure width near 6.0‚Äì6.6 in so it embeds 1:1 at PDF text width (6.5 in at 1 in margins).

## Adding a new note

1. Drop a new `notes/<topic>.md` file. Match the YAML frontmatter style used by `ram_model.md` (title, subtitle, author, fonts, table of contents).
2. Run `make pdf SRC=<topic>` to verify the PDF compiles cleanly.
3. Once it builds, `make all` will pick it up automatically ‚Äî `make list` shows every source/PDF pair the Makefile knows about.
4. If your note embeds figures, follow the figure pipeline above.

## How this material was created

The current state of `ram_model.md` reflects an iterative pass with Claude Code in the loop. The work happened in roughly four phases, each captured in commit history:

1. **Theory pass.** Drafted Parts I‚ÄìV as a graduate-level reference (CLRS-style depth, rigorous definitions, full proofs).
2. **Visual pass.** Replaced every ASCII diagram with a matplotlib-generated figure, infrastructure in `scripts/figures/` (palette and style sharing via `_style.py`, one script per figure). 24 figures total.
3. **Pedagogical pass.** Added scaffolding for first-time readers without disturbing the existing rigor: the ¬ß1.2 quickstart, the ¬ß13 worked-example-first refactor, the ¬ß14.9 pattern dictionary, the ¬ß14.10 Python-cost cheat sheet, the ¬ß16 reorder (visual method first), and the ¬ßA *Test your intuition* sidebars.
4. **Exercise pass.** Added Annexes ¬ßB (60 worked exercises) and ¬ßC (full solutions), distributed across the five Parts and tagged by difficulty.

Each phase was a separate commit; see `git log --oneline notes/ram_model.md` for the trail.
